# Технический документ: офлайн P2P mesh-мессенджер с уровнями доступа

## 1) Цели и ограничения

### 1.1 Цель продукта
Построить мессенджер, который:
- работает без интернета в локальной зоне (Bluetooth/Wi-Fi/Wi-Fi Direct через Nearby Connections);
- поддерживает mesh/store-and-forward (узлы ретранслируют пакеты);
- реализует уровни доступа (0 - максимум прав, больше число - меньше прав);
- допускает развертывание в разных доменах: город, офис, учреждение, площадка мероприятия.

### 1.2 Нефункциональные требования
- Время вывода MVP: 2 дня (хакатон) + возможность последующего развития.
- Android-first (Kotlin), минимум 3 устройства в демо.
- Безопасность: шифрование контента, подписи пакетов, проверка полномочий.
- Устойчивость к обрывам: доставка при повторном контакте.

### 1.3 Что не делаем в MVP
- Звонки/видео.
- Сложные распределенные консенсусы.
- Полная анонимность метаданных (скрываем контент, но не весь трафик).

---

## 2) Варианты продукта (где применимо)

### 2.1 Город/ЧС
Роли: координационный центр, службы, районные группы, волонтеры.  
Уровни:  
- level 0: штаб;
- level 1: службы;
- level 2: старшие групп;
- level 3: исполнители.

### 2.2 Офис/предприятие
Роли: администрация, ИБ, руководители, сотрудники, подрядчики.  
Каналы: общий + служебные сегменты.

### 2.3 Учреждения (кампусы, школы, госплощадки)
Роли: администрация, преподаватели/кураторы, персонал, учащиеся.

Единая модель: одна физическая mesh-сеть + криптографическая сегментация контента по политике доступа.

## 3) Общая архитектура

## 3.1 Компоненты
- **Mobile Node (Android app):**
  - транспорт P2P;
  - локальное хранилище;
  - шифрование/подписи;
  - обработчик политик доступа;
  - ретранслятор пакетов.
- **Admin Console (может быть отдельный Android-режим или desktop tool):**
  - выпуск/отзыв сертификатов участников;
  - назначение уровня и ролей;
  - ротация ключей групп.
- **Policy CA (офлайн доверенный ключ):**
  - корневой ключ подписи полномочий;
  - подпись сертификатов пользователей/устройств.

### 3.2 Сетевой режим
- Без интернета: Nearby Connections.
- С интернетом (опционально): бэкап-синхронизация на сервер.

### 3.3 Базовый поток
1. Узлы обнаруживают соседей.
2. Устанавливают безопасную сессию.
3. Обмениваются digest'ами сообщений (каких пакетов не хватает).
4. Досылают недостающие сообщения.
5. Ретранслируют дальше до TTL.

## 4) Модель безопасности и уровни доступа

### 4.1 Идентичность
У каждого участника:
- `deviceKeyPair` (Ed25519/X25519);
- `userCert` (подписанный админом сертификат):
  - `userId`;
  - `deviceId`;
  - `level`;
  - `roles[]`;
  - `orgId`;
  - `validFrom`, `validTo`;
  - `issuerSignature`.

Клиент хранит публичный корневой ключ CA (`rootPublicKey`) и проверяет подпись сертификатов.

### 4.2 Политика чтения
В сообщении задается `accessPolicy`:
- `ALL` - для всех участников сети;
- `MAX_LEVEL(N)` - читать могут пользователи с `level <= N`;
- `MIN_LEVEL(N)` - читать могут пользователи с `level >= N`;
- `LEVEL_RANGE(min, max)` - читать могут пользователи с `min <= level <= max`;
- `ROLE_ANY([r1, r2])` - хотя бы одна роль;
- `ROLE_ALL([r1, r2])` - все роли;
- `EXPLICIT_USERS([id...])` - точечная рассылка;
- `ORG_INCLUDE([orgId...])` - включить только эти организации;
- `ORG_EXCLUDE([orgId...])` - исключить эти организации.

Для MVP достаточно `ALL` + `MAX_LEVEL(N)` + `ORG_INCLUDE([localOrgId])`.
Для расширенного MVP (если успеваете): добавить `MIN_LEVEL(N)` и `ORG_EXCLUDE`.

### 4.3 Криптомодель (практичная для хакатона)
1. Генерируется случайный `messageKey` (AES-256-GCM).
2. Контент шифруется `messageKey`.
3. Для каждого разрешенного получателя шифруется `messageKey` его публичным ключом (ECIES/X25519 box).
4. Пакет подписывается приватным ключом отправителя.

Итог: любой узел может переслать пакет, но читать сможет только авторизованный получатель.

### 4.4 Ротация и отзыв
- При смене уровня/увольнении: выпуск нового сертификата, старый в CRL.
- Для групповых ключей: `groupKeyVersion++`.
- В MVP отзыв распространяется через подписанные control-пакеты.

## 5) Протокол сообщений (wire format)

Рекомендуемый формат: protobuf (или CBOR/JSON для старта).  
Ниже логическая схема:

```text
Envelope {
  version: uint8
  msgId: UUID
  orgId: string
  senderUserId: string
  senderDeviceId: string
  timestampMs: int64
  ttl: uint8
  hopCount: uint8
  priority: enum {LOW, NORMAL, HIGH, SOS}
  type: enum {CHAT, ACK, CONTROL, KEY_UPDATE}
  accessPolicy: Policy
  payloadCiphertext: bytes
  recipientKeyBoxes: repeated KeyBox
  signature: bytes
}

Policy {
  mode: enum {
    ALL,
    MAX_LEVEL,
    MIN_LEVEL,
    LEVEL_RANGE,
    ROLE_ANY,
    ROLE_ALL,
    EXPLICIT_USERS,
    ORG_INCLUDE,
    ORG_EXCLUDE
  }
  maxLevel: optional uint8
  minLevel: optional uint8
  roles: repeated string
  explicitUsers: repeated string
  orgInclude: repeated string
  orgExclude: repeated string
}

KeyBox {
  recipientUserId: string
  encryptedMessageKey: bytes
}
```

### 5.1 Правила маршрутизации
- Если `msgId` уже видели -> discard (dedup).
- Если `ttl == 0` -> не форвардить.
- Иначе `ttl--`, `hopCount++`, форвард соседям кроме источника.
- ACK только отправителю и/или по обратному пути.

Примечание по организациям: узел может форвардить пакет даже если его `orgId` не входит в политику; ограничение применяется к чтению, а не к ретрансляции.

### 5.2 Надежность
- Локальная очередь отправки.
- Повторы с backoff.
- ACK с таймаутом.
- Digest-синхронизация при новом контакте.

## 7) Экранная модель приложения

Минимум 6 экранов:
1. **Онбординг/идентификация**: импорт сертификата (QR/файл).
2. **Соседи (Peers)**: кто рядом, качество канала, online/offline.
3. **Чаты**: список диалогов/каналов.
4. **Экран чата**: отправка текста, выбор политики доступа (`ALL` или `<= level N`).
5. **Сеть (Network Graph)**: визуализация узлов, hops, pending.
6. **Админ (опционально в MVP-lite)**: выдача сертификата, назначение уровня.

### UX-правила для уровней
- При отправке сразу показывать: "Доступ: уровни 0..N".
- У получателя без прав сообщение отображать как "Недостаточно прав/зашифровано".
- В логах фиксировать только факт доставки, без раскрытия контента.

### UX-правила для организаций
- В форме отправки добавить блок "Организации":
  - `все организации`;
  - `только список` (`ORG_INCLUDE`);
  - `кроме списка` (`ORG_EXCLUDE`).
- Отображать итоговую политику в человекочитаемом виде до отправки:
  - "Получат: org A, org B; уровни <= 2".

## 8) Как поднимать сеть "кем угодно" (город/офис/учреждение)

### 8.1 Модель "Организация"
Каждая сеть = отдельный `orgId` со своим root key.

### 8.2 Шаги запуска новой сети
1. Создать `orgId`.
2. Сгенерировать `root key pair` (офлайн, на доверенном устройстве).
3. Сформировать политику уровней (например, 0..5) и словарь ролей.
4. Выдать админские сертификаты.
5. Подключать участников через выпуск сертификатов (QR provisioning).

### 8.3 Шаблоны политик
- **Город:** уровень + роль (службы/районы).
- **Офис:** уровень + отдел.
- **Учреждение:** уровень + корпус/группа.

### 8.4 Мульти-организация
Один клиент может хранить несколько `orgId`, но трафик и ключи разделяются строго.

## 9) API и внутренние интерфейсы (контракты)

```kotlin
interface TransportService {
    fun startDiscovery()
    fun stopDiscovery()
    fun send(peerId: String, bytes: ByteArray): Result<Unit>
    val incomingPackets: Flow<Packet>
    val peers: Flow<List<PeerState>>
}

interface CryptoService {
    fun sign(data: ByteArray): ByteArray
    fun verify(data: ByteArray, sig: ByteArray, pubKey: ByteArray): Boolean
    fun encryptPayload(plain: ByteArray, key: ByteArray): ByteArray
    fun decryptPayload(cipher: ByteArray, key: ByteArray): ByteArray
    fun encryptMessageKeyForRecipient(key: ByteArray, recipientPub: ByteArray): ByteArray
}

interface PolicyEngine {
    fun canRead(policy: AccessPolicy, cert: UserCert): Boolean
    fun resolveRecipients(policy: AccessPolicy, directory: List<UserCert>): List<UserCert>
}
```

Порядок фильтрации в `resolveRecipients(...)`:
1. Взять активные сертификаты из directory.
2. Применить фильтр организаций (`ORG_INCLUDE/ORG_EXCLUDE`).
3. Применить фильтр уровней (`MAX_LEVEL`, `MIN_LEVEL`, `LEVEL_RANGE`).
4. Применить ролевые/точечные фильтры (`ROLE_*`, `EXPLICIT_USERS`).
5. Вернуть итоговый список получателей для генерации `KeyBox`.

## 10) Алгоритмы (MVP)

### 10.1 Отправка сообщения
1. Пользователь вводит текст и выбирает политику.
2. `PolicyEngine.resolveRecipients(...)`.
3. Генерация `messageKey`.
4. Шифрование payload (`AES-GCM`).
5. Формирование `KeyBox` на каждого разрешенного получателя.
6. Подпись `Envelope`.
7. Сохранение в Room (status = PENDING).
8. Передача всем соседям + ретраи до ACK/timeout.

Псевдологика фильтрации (для команды):

```kotlin
fun matchesPolicy(user: UserCert, policy: AccessPolicy): Boolean {
    if (!user.isActiveNow()) return false

    if (policy.orgInclude.isNotEmpty() && user.orgId !in policy.orgInclude) return false
    if (user.orgId in policy.orgExclude) return false

    policy.maxLevel?.let { if (user.level > it) return false } // <= N
    policy.minLevel?.let { if (user.level < it) return false } // >= N

    if (policy.mode == LEVEL_RANGE) {
        val min = policy.minLevel ?: 0
        val max = policy.maxLevel ?: 255
        if (user.level < min || user.level > max) return false
    }

    return true
}
```

### 10.2 Прием сообщения
1. Проверить подпись и сертификат отправителя.
2. Проверить dedup (`msgId`).
3. Если есть свой `KeyBox`:
   - расшифровать `messageKey`;
   - расшифровать payload;
   - показать в чате.
4. Отправить ACK.
5. Если `ttl > 0` - форвард дальше.

### 10.3 Обновление прав
1. Админ выпускает новый сертификат.
2. Подписанный control-пакет распространяется по сети.
3. Клиенты обновляют directory/CRL.
4. Новые сообщения шифруются по актуальным правам.

## 12) Демо-сценарий для жюри (3-4 минуты)

1. Интернет отключен на всех устройствах.
2. Устройства A, B, C; A не видит C напрямую.
3. Пользователь level 1 отправляет сообщение с политикой `<=1`.
4. Выбраны организации-получатели (`ORG_INCLUDE: [org-city, org-med]`).
5. B ретранслирует, C (level 2 или не из нужной org) получает пакет, но не читает контент.
6. Устройство level 0 из разрешенной org читает сообщение успешно.
7. Показать журнал hops/доставки и сетевой граф.

---

## 13) Риски и как закрыть

- **Проблемы Nearby на конкретных девайсах** -> иметь 3-4 тестовых телефона разных моделей.
- **Падение батареи/фоновые ограничения** -> foreground service в демо-режиме.
- **Сложность крипты** -> не усложнять, использовать проверенные библиотеки, минимальный набор примитивов.
- **Нестабильность маршрутизации** -> ограниченный TTL, простая flooding-модель.

## 14) Безопасность и соответствие

- Хранить приватные ключи в Android Keystore.
- Пины/биометрия на вход в приложение.
- Подпись всех control-сообщений (ротация, отзыв, уровни).
- Логи без открытого payload.
- Документировать ограничение: метаданные трафика частично наблюдаемы.
