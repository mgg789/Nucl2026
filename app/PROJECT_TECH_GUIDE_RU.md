---

## 1. Что это за приложение

Это Android-прототип офлайн-мессенджера для mesh/P2P сети:
- связь между устройствами без интернета через Google Nearby Connections;
- отправка сообщений с политикой доступа:
  - по уровням (`<= N`, `>= N`);
  - по организациям (`include`, `exclude`);
- шифрование содержимого сообщения;
- подпись каждого сообщения ключом устройства (Android Keystore);
- подтверждение доставки (ACK), повторные отправки (retry), дедупликация и TTL форвардинг.

---

## 2. Стек технологий (что и зачем)

- `Kotlin` — основной язык.
- `Jetpack Compose + Material3` — UI.
- `Google Nearby Connections API` — p2p обмен payload между устройствами рядом.
- `Android Keystore` — генерация/хранение ключей устройства и подпись.
- `AES-GCM` — шифрование payload.
- `SharedPreferences` (через `MessageQueueStore`) — хранение статусов исходящих сообщений между перезапусками.

---

## 3. Словарь терминов (терминология проекта)

- `Peer` — соседнее устройство в Nearby-сети.
- `Mesh` — сеть узлов, где узлы могут ретранслировать сообщения дальше.
- `Envelope` — транспортный пакет сообщения (метаданные + зашифрованный payload + подпись).
- `AccessPolicy` — политика доступа к чтению.
- `ACK` — подтверждение получения сообщения.
- `TTL` — число хопов, сколько раз сообщение можно форвардить.
- `hopCount` — сколько ретрансляций уже прошло.
- `dedup` — отсечение дубликатов по `messageId`.
- `Keystore key` — приватный ключ устройства в Android Keystore.

## 4. Карта файлов (кратко, что делает каждый)

## 4.1 Точка входа

- `app/src/main/java/com/example/app/MainActivity.kt`  
  Запускает Compose UI и рендерит `MeshAccessScreen`.

## 4.2 UI слой

- `app/src/main/java/com/example/app/ui/MeshAccessScreen.kt`
  - главный экран прототипа;
  - выбор локальной identity (кто это устройство);
  - настройка политики доступа (уровни, include/exclude org);
  - кнопки сети (`Старт Nearby`, `Стоп`);
  - demo-тоггл `ACK drop`;
  - отправка сообщений;
  - визуализация входящих, сетевых логов, топологии и статусов доставки.

## 4.3 Domain слой (протокол и правила)

- `app/src/main/java/com/example/app/domain/AccessPolicy.kt`
  - модель политики доступа;
  - `PolicyEngine.matches(...)` и `resolveRecipients(...)`.

- `app/src/main/java/com/example/app/domain/MeshModels.kt`
  - `MeshEnvelope` (формат пакета);
  - `MeshMessageType` (`CHAT`, `ACK`);
  - сериализация/десериализация JSON;
  - `MeshCrypto` (AES-GCM encrypt/decrypt);
  - `MeshSignature` (verify подписи);
  - фабрики `createChat`, `createAck`;
  - `SampleDirectory` и `LocalIdentity` для demo-каталога пользователей.

## 4.4 Data слой (сеть, подпись, очередь)

- `app/src/main/java/com/example/app/data/NearbyMeshClient.kt`
  - управление Nearby (advertise, discover, connect);
  - отправка/прием payload;
  - verify подписи входящих;
  - decrypt payload;
  - ACK/retry/timeout;
  - дедупликация и форвардинг по TTL;
  - сбор метрик (`MeshStats`);
  - работа с очередью статусов доставки.

- `app/src/main/java/com/example/app/data/DeviceKeyStoreSigner.kt`
  - выбирает alias по `userId`;
  - генерирует EC ключ при отсутствии;
  - подписывает строку `SHA256withECDSA`;
  - отдает публичный ключ (Base64).

- `app/src/main/java/com/example/app/data/MessageQueueStore.kt`
  - хранит исходящие сообщения и состояние доставки;
  - состояния: `QUEUED`, `SENT`, `ACKED`, `FAILED`;
  - хранение в `SharedPreferences`, восстановление после перезапуска.

## 4.5 Конфиги

- `app/app/build.gradle.kts` — compile/target SDK, зависимости модуля.
- `app/gradle/libs.versions.toml` — централизованные версии зависимостей.
- `app/src/main/AndroidManifest.xml` — разрешения Bluetooth/Location/Wi-Fi и activity.

## 5. Как работает система: end-to-end путь

## 5.1 Старт приложения

1. Запускается `MainActivity`.
2. Открывается `MeshAccessScreen`.
3. UI запрашивает runtime permissions (`BLUETOOTH_*`, `ACCESS_FINE_LOCATION`).
4. Пользователь выбирает локальный профиль (`LocalIdentity`).

## 5.2 Старт сети

1. Нажимается `Старт Nearby`.
2. `NearbyMeshClient.start(identity)`:
   - поднимает advertising;
   - запускает discovery;
   - принимает/инициирует подключения.
3. В `DeviceKeyStoreSigner` выбирается/создается ключ под `userId`.

## 5.3 Отправка сообщения

1. Пользователь вводит текст и политику (`AccessPolicy`).
2. UI вызывает `nearbyClient.buildChatEnvelope(...)`.
3. В `MeshEnvelope.createChat(...)`:
   - текст шифруется AES-GCM;
   - собирается пакет с policy, ttl, sender metadata.
4. Пакет подписывается приватным ключом устройства (`DeviceKeyStoreSigner.sign(...)`).
5. `sendChat(...)`:
   - пишет запись в очередь (`QUEUED`);
   - отправляет в подключенные endpoints (`SENT`);
   - добавляет в pending ACK;
   - запускает retry-цикл.

## 5.4 Прием сообщения

1. Приходит payload из Nearby.
2. Парсинг `MeshEnvelope.fromJsonBytes(...)`.
3. Проверка подписи `MeshSignature.verify(...)`.
4. Проверка dedup по `messageId`.
5. Расшифровка payload (AES-GCM).
6. Отправка ACK (если не включен demo `ACK drop`).
7. При `ttl > 0` — форвардинг дальше:
   - `ttl = ttl - 1`;
   - `hopCount = hopCount + 1`.
8. UI показывает входящее; право чтения проверяется через `PolicyEngine.matches(...)`.

## 5.5 Обработка ACK и retries

1. Если ACK пришел:
   - сообщение удаляется из pending;
   - статус в очереди -> `ACKED`.
2. Если ACK не пришел:
   - выполняются ретраи (интервал ~2.5 сек, до лимита);
   - после лимита -> `FAILED`.

## 6. Логика доступа (уровни и организации)

Политика в текущем прототипе:
- `maxLevel` (`<= N`);
- `minLevel` (`>= N`);
- `includeOrgs` (получают только эти org);
- `excludeOrgs` (эти org исключаются).

Правило `PolicyEngine.matches(user, policy)`:
1. пользователь активен;
2. если include задан — `user.orgId` должен входить;
3. `user.orgId` не должен быть в exclude;
4. уровень проходит `maxLevel/minLevel`.

Важно: сейчас это **фильтр чтения** на клиенте + симуляция получателей по каталогу.  
Сетевая отправка физически идет подключенным peer'ам Nearby.

## 7. Безопасность: что уже есть и что пока demo-уровень

Уже есть:
- шифрование содержимого (AES-GCM);
- подпись сообщений ключом из Android Keystore;
- проверка подписи на входящих;
- отсутствие plaintext в payload.

Ограничения текущего прототипа:
- симметричный ключ AES общий для demo, не пер-пользователь;
- нет PKI/сертификатов доверенного центра;
- нет ротации/отзыва ключей;
- нет полноценной E2E раздачи message key per recipient.

## 11. QR-выдача "вживую" (на будущее, вне хакатона)

Что это такое:
- администратор вживую выдает участнику права через QR;
- участник сканирует QR и получает `userId/orgId/level/roles` + подписанный сертификат;
- после импорта приложение начинает доверять этой identity для политики доступа.

Зачем это нужно:
- быстрое подключение новых участников на месте (город, штаб, мероприятие, офис);
- исключение ручного ввода уровней и org на каждом устройстве;
- контроль полномочий через подпись админского/root ключа.

Текущий статус:
- в текущем прототипе **это не реализовано**;
- сейчас identity выбирается локально из demo-каталога (`SampleDirectory`).

Целевая реализация (пост-хакатон):
1. **Admin mode**: экран создания участника и назначения `level/org/roles`.
2. **Выпуск credential**: формирование JSON сертификата + подпись root/private key.
3. **QR encode**: упаковка credential в QR (или в несколько QR при большом размере).
4. **Client import**: скан QR, валидация подписи root public key, сохранение cert в локальное хранилище.
5. **Enforcement**: сверка `senderUserId <-> public key <-> cert` при приеме сообщений.
6. **Revocation/rotation**: отзыв и ротация прав через control-пакеты.

Минимальный формат credential для QR:
- `userId`
- `orgId`
- `level`
- `roles[]`
- `validFrom`, `validTo`
- `devicePublicKey` (или binding token)
- `issuerId`
- `signature`

Рекомендуемые библиотеки для реализации:
- генерация QR: `zxing-core`, `zxing-android-embedded`;
- сканирование QR: `ML Kit Barcode Scanning` или `zxing`;
- подпись/валидация: существующий Keystore + отдельный root key (админский контур).
